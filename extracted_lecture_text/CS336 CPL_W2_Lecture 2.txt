LANGUAGE DESIGN: EFFICIENCY 

--- PAGE 14 ---
▪ A language is efficient if it allows programs to run 
fast, use little memory, and be written quickly. These 
three goals often conflict with each other.
▪ Driving a manual car (C) = maximum fuel efficiency, 
full control. Driving an automatic (Python) = easier 
and faster to learn, but less control over performance.
14
LANGUAGE DESIGN: EFFICIENCY 

--- PAGE 15 ---
▪ Language designers nearly always claim that their 
new languages support efficient programs, but what 
does that really mean? 
▪ Language designers usually think of the efficiency of 
the target code first. 
▪ That is, they strive for a language design that allows 
a translator to generate efficient executable code.

--- PAGE 18 ---
▪ In Java code, since the data types of variables are 
known at compile time, the compiler ensures that only 
valid integer and string operations are performed.
 
▪ As a result, the runtime system does not need to check 
data types during execution.
18
LANGUAGE DESIGN: EFFICIENCY 

--- PAGE 19 ---
▪ In contrast, the equivalent code in Python simply 
assigns values to typeless variables.
▪  The absence of a type specification at compile time 
forces the runtime system to check the type of a 
Python variable’s value before executing any 
operations on it. 
▪ This causes execution to proceed more slowly than it 
would if the language were statically typed.

Language Design: Security
▪ A secure language makes it harder to write buggy or 
dangerous code.
▪ Core principle: Detect errors as early as possible, ideally 
at compile time, not at runtime, and never silently.
ANALOGY
Spell-check as you type (compile-time) vs. finding 
typos after you've sent the email (runtime). 
Better still: grammar that makes the error 
impossible to write.
WHY IT MATTERS
Silent errors ,wrong results with no crash  are the 
costliest bugs. Security-focused design forces errors 
to surface as early as possible in development.

Language Design: Security
▪ Aliasing: two names refer to the same memory. Changing 
one changes the other….. silently.
The aliasing problem in C
int x = 10;
int *p = &x;  // p aliases x
*p = 99;      // x is now 99
// x changed — but where?
Two variables refer to the same memory location. 
Modifying one silently affects the other , a rich source 
of hard-to-find bugs in C programs.
How languages address this
C No restrictions on pointers — full aliasing allowed
Java No pointer arithmetic; aliasing exists for objects 
but is controlled
Python Assignment creates references; mutable vs. 
immutable types clarify behavior
Haskell Immutable values by default — aliasing is 
harmless because data never changes
Ada Alias detection is part of the type system; 
restricted aliasing constructs

Language Design: Extensibility 
▪ A language is extensible when programmers can add new 
features using the same mechanisms as built-in ones.
▪ Ideal extensibility: no visible difference between what 
the language gives you built-in and what you define 
yourself.
ANALOGY
A smartphone with an app store = extensible. 
A calculator with fixed buttons = not extensible.
You expand what it does without changing the 
device itself.
THE IDEAL
If someone reading code can't tell whether '+' was 
built into the language or defined by the 
programmer, the language has achieved full 
extensibility.

▪ Lets take a closer look at operator overloading
Can you make  a + b  work for your own type?
Haskell
Highest
Can overload existing operators AND 
define brand-new ones (e.g. <+>). 
Operators are functions; full extensibility.
data Vec = Vec Int Int
(+) (Vec a b) (Vec c d) = Vec (a+c) 
(b+d)
Python
High
Overload via __add__, __eq__, __lt__

class Vec:

C++
Medium
Overloading supported, but cannot 
create new operators. Some built-in 
behavior (e.g. short-circuit &&) cannot

Vec operator+(Vec a, Vec b) {
  return Vec{a.x+b.x, a.y+b.y};

Java
None
No operator overloading at all. Must use 
methods: a.add(b) rather than a + b. User

Case Study: C++ vs Python
▪ Many C++ irregularities were not accidental they were the 
unavoidable cost of C compatibility.
Violates: uniformitySemicolon after class, not function
C struct declarations allowed variables after the closing 
brace. C++ kept this for compatibility. Structures and 
classes reuse the same syntax.
class A { ... };  // required
void f() { ... }  // FORBIDDEN
Violates: orthogonalityArrays differ from all other types
C treats arrays as pointer decay. C++ inherited this. You 
cannot return an array from a function, assign arrays with 
=, or pass them by value.
int a[5] = {1,2,3,4,5};
int b[5] = a;  // ERROR
b = a;         // ERROR
Violates: generalityNo nested function definitions
C uses a simple stack-based runtime. Nested functions 
would require closures and a more complex activation 
record — too expensive for C's goals.
void outer() {
  void inner() { ... }  // illegal!
}

Case Study: C++ vs Python
▪ Python design goal: maximum readability and programmer 
productivity, with a highly regular, object-oriented design.
Regularity Very high
Everything is an object, no primitive/object split. 
Functions are first-class. All types behave the same 
way.
Extensibility Very high
Operator overloading via dunder methods 
(__add__, __eq__). Metaclasses allow radical 
customisation.
Prog. efficiency Very high
Concise syntax, list comprehensions, dynamic 
typing, batteries included. Developer productivity 
is top priority.
Security Medium
Dynamic typing means type errors surface at 
runtime, not compile time. No type checker by 
default (though mypy helps).
Exec. efficiency Lower
Interpreted, dynamically typed  generally 10–100× 
slower than C for compute-heavy tasks. Traded for 
writability.

Case Study: C++ vs Python
▪ Why such design decisions chosen for python?
Everything is an object
Integers, strings, functions, classes  all are objects. No 
primitive/reference split (unlike Java). This maximises orthogonality: 
one model, no exceptions.
+ Complete regularity , same rules for everything
- Small overhead for numeric computations vs. raw int
Dynamic typing
Types are checked at runtime, not compile time. A variable can hold 
any type at any time. This reduces boilerplate and improves writability 
dramatically.
+ Faster to write, more flexible, easier to prototype
- Type errors become runtime crashes, not compile 
errors
Indentation as syntax
Python uses indentation to define code blocks instead of braces. This 
enforces readable code and eliminates one class of inconsistency 
("should I use braces?").
+ Uniform visual structure; readable code forced by 
design
- Unusual  mixing tabs and spaces causes cryptic 
errors
